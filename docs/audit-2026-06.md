# Program audit — June 2026

A full-stack audit (architecture, background work, relationships, processes) with web-researched
upgrade opportunities and a prioritized backlog. Companion to [roadmap.md](roadmap.md).

## Overall health
Mature and broad: a Vite+TS viewer (three/@thatopen), a 74-module config-driven GC portal with
workflows/attachments/revisions, a development proforma, IFC→Fragments pipeline, and (recently)
multi-format reference models + GIS, federated clash, PDF takeoff, contract lifecycle + PAdES
e-signatures, and a Report Center. Test gate: 35 backend suites + 24 web tests, CI + Pages green.
Fundamentals (mobile/PWA, RFI/submittal depth, cost tracking, integrations) are solid — which the
2026 research says to confirm *before* chasing AI features.

## Background work — do we need Celery?
**Current:** the only genuinely long task is **IFC→Fragments conversion** (`_publish_bg`,
`routers/authoring.py:304`) — a daemon thread + a `publish_status.json` the client polls. Connector
auto-sync is an `asyncio` loop (`main.py`); Redis is an *optional* dependency (rate-limiting only).

**Recommendation: no Celery now.** For the self-hosted / desktop single-project mission, the
thread + status-poll is the right amount of machinery (no broker to run, works offline). Celery's
value — durable queue, retries, horizontal workers, beat scheduling — only pays off at a **multi-tenant
cloud tier with concurrent heavy conversions**. Its weaknesses today: a publish lost on restart, no
retry, single-instance only.

**When/what instead:** when a cloud tier with real conversion load arrives, add a task queue — and
since `redis` is already optional here, **RQ or Dramatiq on Redis is the better fit than Celery**
(far less ceremony; no separate result backend needed). Low-cost prep now: keep `_publish(pid)` a
pure function (it already is) so it can be invoked by either the thread *or* a worker, and persist
publish status in the DB (not just a JSON blob) so it survives restarts. → backlog item.

## Relationships — strong; few gaps
The reference graph is extensive and the change chain is complete
(`rfi → change_event → pco_request → cor/proposal`, with portal **conversions** auto-linking them).
Budget/cost ties run through `cost_code` (budget, commitment, subcontract, sov, direct_cost,
sub_invoice, staffing, timesheet, schedule_activity…). Gaps worth closing (small):
- **Approved/executed COR → SOV/billing line.** COs flow to the *revised GMP* (computed) but don't
  spawn an explicit SOV/billing line; add a conversion `cor (executed) ⤳ sov line`.
- **Awarded bid → subcontract + commitment.** `subcontract → bid_submission` exists, but there's no
  one-click "award `bid_submission` ⤳ create subcontract (+commitment)" conversion.
- **cor → change_event** is only indirect (via `pco_request`); a direct optional ref aids traceability.

## Process efficiency
- **Conversions** already automate the change chain + QA chain (good). Extend to the two ties above.
- **Blocking `prompt()`/`confirm()`** remain in a few data-entry spots (AI-estimate description,
  add-discipline name) — replace with the existing `modalShell` for a modern, non-blocking UX. Most
  `confirm()` uses are legitimate destructive guards (keep).
- **Reports/exports** now unified (Report Center). Estimates → GAEB/Excel. Good.

## Web-researched upgrade opportunities (2026)
Sources favor "fundamentals first, AI as a tiebreaker." We have the fundamentals + an Anthropic seam
(`ai.py`), so the high-ROI AI adds are:
1. **RFI triage** — auto-categorize incoming RFIs (discipline/type/urgency) + suggested ball-in-court
   and a draft response (we already have `draft_rfi`; add triage). *High value, low risk.*
2. **Predictive schedule alerts** — flag late predecessors, slipping milestones, procurement/lead-time
   gaps from the CPM + lookahead we already compute (rules first, AI narrative second). *Medium.*
3. **Document/risk intelligence** — extend `risk_summary` into a periodic project-risk digest across
   cost + schedule + open items. *Medium.*
4. **AI schedule optimization** (re-sequence critical path) — large; evaluate later.

## Prioritized backlog
1. Relationship/process ties: COR→SOV line + awarded-bid→subcontract conversions (low risk).
2. AI RFI triage (reuses the AI seam).
3. Replace data-entry `prompt()`s with modals (UX polish).
4. Durable publish status (DB) + keep `_publish` worker-callable (queue-readiness; no Celery).
5. Predictive schedule alerts; risk digest.
6. (Deferred) cloud task queue via RQ/Dramatiq+Redis; AI schedule optimization.

Out of scope / deliberately not done: Celery (see above); chasing unverified AI accuracy claims.
